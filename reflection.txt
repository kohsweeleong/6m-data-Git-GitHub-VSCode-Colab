## Q1. The "Snapshot" Workflow
One thing I now understand is that saving work in Git isn't a single step like clicking "Save" in a Word document; it’s a three-tier progression. I could explain to a friend that Git works like a professional photo shoot.

First, you have your Working Directory where you’re actively making a mess (coding). 
When you’re ready, you "Stage" your files (the + icon), 
which is like posing your subjects for the camera. 
Then, you "Commit," which is like snapping the photo—it creates a permanent, 
time-stamped version of that exact moment. Finally, 
"Pushing" is like uploading that photo to an online gallery (GitHub) so others can see it.
Understanding this distinction helps me see that my code isn't truly 
"saved" in my history until I’ve completed the full loop.

To figure this out, my plan is to intentionally create a conflict. 
I will edit a file directly on the GitHub website and then try to
